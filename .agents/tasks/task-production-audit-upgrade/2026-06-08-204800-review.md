# Production audit and feature upgrade across four priority tiers

This change layers security hardening, operational improvements, an audit-trail system, and three new AI-driven features onto an Express + Drizzle + Postgres API server. FEAT-001 adds ownership enforcement on CRUD routes, fixes a CORS fail-open in production, validates prompt length, adds graceful shutdown, and tightens the Stripe webhook. FEAT-002 introduces pagination, database indexes, SSE heartbeats on streaming endpoints, Docker layer-cache optimization, configurable AI models via env vars, and connection-pool limits. FEAT-003 adds `generation_runs` and `project_revisions` tables with Drizzle schema, a recording service, and two new history/runs endpoints. FEAT-004 introduces a style-presets system, Gemini-based icon generation, and a Gemini-vision visual-feedback endpoint.

**Watch for:** Ownership enforcement is incomplete—`approve-plan`, `answer-clarifications`, and `generate-web` routes still allow any user to act on any project (confirmed). The `visual-feedback` endpoint accepts unbounded base64 payloads with no body-size cap beyond the default Express 100 KB limit, which either makes the feature non-functional for real screenshots or (if overridden elsewhere) creates a DoS vector (likely). The audit-trail writes are fire-and-forget with swallowed errors, so generation_runs may silently never record failures (confirmed).

## High-level view

Ownership enforcement was applied uniformly to CRUD, files, share, download, and preview routes using a "project not found" 404 pattern that avoids leaking existence. The three highest-traffic write paths—`approve-plan`, `answer-clarifications`, and `generate-web`—were not upgraded and remain unprotected, meaning any authenticated user who knows a project ID can trigger a build or mutate another user's project state.

The generation-history service records plan and build events via non-blocking `.catch(() => {})` calls. The empty catches mean persistent insert failures (missing table, connection exhaustion) produce no signal—audit data silently disappears.

The visual-feedback endpoint sends up to 15 KB of source context plus a full screenshot to Gemini, but the request schema only validates that `screenshot` is a non-empty string—no max-length constraint. Express's default 100 KB body limit means real screenshots (200 KB–2 MB base64) will be rejected with 413 before reaching the handler.

No SQL migration adds the `style_preset` column to the `projects` table. The Drizzle schema declares it, and it was likely added via `drizzle-kit push` in development, but production deployments running only migration files will fail on insert.

The CORS fix is fail-closed when `ALLOWED_ORIGINS` is empty in production, but the `!origin` short-circuit still allows requests with no `Origin` header through unconditionally—standard for server-to-server calls but exploitable for API abuse from tools that strip the header.

1. **Missing ownership on approve-plan, answer-clarifications, generate-web** — Any authenticated user can trigger code generation or mutate status on another user's project. Add the same userId guard used on CRUD routes.
2. **Unbounded screenshot payload** — `visualFeedbackSchema` does not constrain `screenshot` length. Express's default 100 KB `json()` limit likely rejects real screenshots (typically 200 KB–2 MB base64). Either raise the body limit on this route and add a max-length in the schema, or stream the upload.
3. **Ownerless-project access logic blocks legitimate anonymous usage** — The pattern `if (project.userId === null && req.user)` returns 404 for authenticated users viewing their own anonymous projects created before they registered. This may be intentional but is worth verifying.
4. **Fire-and-forget audit writes** — `recordGenerationRun` and `recordProjectRevision` errors are silently swallowed. At minimum, log the failure so missing audit data is observable.
5. **No migration for `style_preset` column** — The Drizzle schema declares `stylePreset: text("style_preset")` on projects but no SQL migration adds the column. Production deployments relying on migrations will break.
6. **CORS origin-less requests** — Requests without an `Origin` header bypass CORS checks entirely. This is HTTP-spec-compliant but allows API abuse from non-browser clients even when ALLOWED_ORIGINS is properly configured.

## Ownership enforcement gaps on generation routes

The CRUD routes (`GET /projects/:id`, `DELETE`, `/files`, `/share`, `/download`, `/preview`, `/history`, `/runs`) and the `POST /generate` endpoint all check `project.userId` against `req.user.id` and return 404 on mismatch. Three routes skip this entirely:

```typescript
// approve-plan: fetches project, checks only for "not found" and "already generating"
const [project] = await db.select().from(projectsTable).where(eq(projectsTable.id, id));
if (!project) { sendEvent({ error: "Project not found" }); res.end(); return; }
// No userId check follows — proceeds directly to generation
```

The same pattern appears in `answer-clarifications` (line ~770) and `generate-web` (line ~30 of web-generate.ts). An attacker only needs a valid project ID (sequential integers starting at 1) to hijack any project's generation pipeline, overwriting its architecture plan, files, and status.

## Screenshot payload size and Express body limit

`express.json()` defaults to a 100 KB body limit. A typical mobile screenshot encoded as base64 is 200 KB–2 MB. The `visual-feedback` endpoint will reject most real screenshots with a 413 status before the route handler ever runs. If the intent is to support real screenshots, the route needs its own body parser with a higher limit:

```typescript
app.use("/api/projects/:id/visual-feedback", express.json({ limit: "5mb" }));
```

And `visualFeedbackSchema` should enforce a ceiling (`z.string().max(7_000_000)` for ~5 MB base64) to prevent memory exhaustion.

## Ownership model asymmetry for ownerless projects

The access pattern rejects authenticated users from accessing projects where `userId` is null:

```typescript
if (project.userId === null && req.user) {
  res.status(404).json({ error: "Project not found" }); return;
}
```

This means if a user creates a project while unauthenticated (no session), then later logs in, they lose access to their own project. The share-token bypass works, but the UX gap is unusual. If this is intentional (force anonymous projects to stay anonymous), it should be documented; if not, consider migrating ownerless projects to the user on first login.

## Missing `style_preset` migration

The Drizzle schema at `lib/db/src/schema/projects.ts` declares:

```typescript
stylePreset: text("style_preset"),
```

Migration 0007 adds indexes; migration 0008 adds `generation_runs` and `project_revisions`. Neither adds `style_preset` to the `projects` table. A production deployment running only SQL migrations will get a column-not-found error on any project insert that includes `stylePreset`.

## Test coverage gaps

76 tests cover ownership isolation (cross-user access returns 404), pagination response shape, prompt length validation, unauthenticated access returns empty lists, style preset retrieval, icon generation returns 503 without Gemini key, and visual feedback validation. Not tested: `approve-plan` and `answer-clarifications` ownership (because they don't have it), `generate-web` ownership, heartbeat behavior, graceful shutdown signal handling.
